# PROBE B — Additive wiring (the `scope` / capability-marker mechanism)

> Isolated spike. Self-contained, references no repo projects, edits nothing
> outside this folder. net10.0 consumer + netstandard2.0 generator, reusing the
> `Microsoft.CodeAnalysis.CSharp` **4.11.0** version from `spike/gen/`.

## Idea

Authored code (D2 "scope form" from `authoring-dialects.md`) wires its
dependencies at the use-site with capability markers:

```csharp
App.Handler<AddItem>(scope =>
{
    var users = scope.Get<Repo<User>>();                       // marker AND real call
    var book  = scope.Ask<BookDetails>(new BookDetailsQuery(default)); // marker AND real call
    var user  = users.Get("ada@example.com");
    user.AddItemToCart(new CartItem(default, 1, book.Price, book.Label));
    users.Save();
    return user;
});
```

`scope.Get<T>()` and `scope.Ask<T>()` are **real runtime methods** that resolve
from a registered container (`consumer/Scope.cs`). The very same call sites are
**markers a classic `IIncrementalGenerator` scans** to emit, *additively*, a
typed dependency manifest and a registration/validation method
(`consumer/gen/WiringGenerator.cs` → `Wiring.g.cs`).

Two things hold at once, with **no interceptors and no `[InterceptsLocation]`**:

1. the lambda **compiles and runs as written** (the body is never edited), and
2. the generator **statically enumerates the dependencies** from the markers and
   emits new code beside the lambda.

## Why

Maps to two invariants in `spike/README.md`:

- **Minimal dialect — in the wiring.** The marker grammar lives in the wiring
  (`scope.Get<T>` / `scope.Ask<T>`), a small regular vocabulary an agent emits
  reliably. Here it's `Get` (service) and `Ask` (query) — the D2 verb core.
- **Source-generation back-fills inline — bounded.** The author declares a need
  at the use-site; the generator mints *only* the DI/plumbing manifest, never new
  behavior.

**The reviewer's doubt this probe answers:** a reviewer warned that to turn a
body call like `Save(u)` into injected I/O you'd have to *rewrite the call* — which
needs fragile **interceptors** (`[InterceptsLocation]`, file+line+column pinned to
the call site, brittle across edits). The honest, fully-additive alternative is a
**real `scope` object** plus a generator that **reads the markers for
registration** instead of rewriting the body. This probe builds exactly that path
and shows it stands up.

The distinction matters:

| | Interceptor path (rejected) | Additive path (this probe) |
|---|---|---|
| Mechanism | rewrites an existing call at a pinned location | emits *new* code next to the lambda |
| Body | edited/redirected by the compiler | untouched — runs as authored |
| Fragility | location attributes break on any edit above the call | none; re-scans syntax every build |
| What `scope.Get<T>` is | a placeholder the interceptor replaces | a real container call **and** a scan target |

## What was proved

- **Compiles?** Yes. `dotnet build -c Release` on both projects: `0 Warning(s), 0 Error(s)`.
- **Runs?** Yes. The authored handler bodies execute against the real container.
- **Generator statically enumerates deps from the markers?** Yes — see the
  emitted `Wiring.g.cs` below (captured verbatim as `Wiring.g.captured.cs`).

became `ProbeB.Repo<ProbeB.User>` (fully-qualified, generic), and `Ask<BookDetails>`

registration kind.


So the static manifest and the runtime container are cross-checked by generated
code — the deps the generator *found* must be the deps the container *has*.

## Key generator snippet

The whole mechanism is a syntax predicate + a marker scan — no `SemanticModel`
rewriting of the body, no location pinning:

```csharp
// 1. find the registration call sites: Handler<T>(scope => …) / Endpoint<T>(scope => …)
var handlers = context.SyntaxProvider.CreateSyntaxProvider(
    predicate: static (n, _) => IsHandlerRegistration(n),
    transform: static (ctx, _) => Extract(ctx));

// 2. inside each lambda, scan invocations shaped `scope.Get<T>()` / `scope.Ask<T>()`
foreach (var call in lambda.DescendantNodes().OfType<InvocationExpressionSyntax>())
{
    if (call.Expression is not MemberAccessExpressionSyntax ma) continue;
    if (ma.Expression is not IdentifierNameSyntax recv || recv.Identifier.ValueText != scopeParam) continue;
    if (ma.Name is not GenericNameSyntax verb || verb.TypeArgumentList.Arguments.Count != 1) { /* dynamic → note */ continue; }
    var kind = verb.Identifier.ValueText switch { "Get" => DepKind.Service, "Ask" => DepKind.Query, _ => DepKind.None };
    var type = ctx.SemanticModel.GetTypeInfo(verb.TypeArgumentList.Arguments[0]).Type; // resolve T semantically
    deps.Add(new Dependency(kind, type.ToDisplayString()));
}

// 3. RegisterSourceOutput: emit the manifest + RegisterDiscovered — purely additive.
```

The lambda is read, never written. There is no `[InterceptsLocation]` anywhere in
the probe (`grep` it).

## Honest limitations

1. **Statically-analyzable only — by design and by enforcement.** The generator
   sees `scope.Get<T>()` where `T` is a concrete type literal in source. It
   *cannot* see a dependency chosen at runtime. We made that a closed door:
   `Scope` has **no** `Get(Type)` / non-generic overload, so a dynamic
   `scope.Get(someRuntimeType)` *won't even compile*. The marker grammar is
   deliberately restricted to the enumerable subset. (If such an overload existed,
   the generator records a `// NOTE: …dynamic…` comment and skips it rather than
   guessing.)
2. **Looped / repeated calls → one dependency.** `scope.Get<Read>()` inside a
   `for` loop is found once; the manifest is "distinct types this handler
   references", not a call count. The `Reindex` handler demonstrates this (calls
   it thrice, discovered once). Correct for *registration*; it says nothing about
   *cardinality*.
3. **Generic type-parameter markers are not resolvable.** `scope.Get<T>()` inside
   a generic method would record the open type parameter `T`, which is useless for
   registration — the call must be monomorphised to a concrete type at the use
   site. Not exercised here (handlers are non-generic); flagged for honesty.
4. **`scope` must be a single, named lambda parameter.** The scan keys on the
   receiver identifier equal to the lambda's parameter name. Aliasing the scope to
   a local (`var s = scope; s.Get<…>()`) would currently be missed. Tightening
   this would need light data-flow; out of scope for the probe.
5. **Diagnostics are minimal.** The probe emits no `Diagnostic` descriptors (no
   squiggles for an unresolved marker); the wiring-gap check is a *runtime*
   `RegisterDiscovered` throw, not a build error. A production version would
   promote the gap to a compile-time `Diagnostic` and could emit per-handler
   strongly-typed accessors instead of a stringly manifest.
6. **The manifest is stringly-typed** (`Dep(string Kind, string Type)`). Adequate
   to prove enumeration; a real version would emit typed `Resolve` calls or a
   typed dependency record per handler so consumers bind without string matching.

## Verdict

The additive path is **real and not a workaround**: you genuinely get *both* "the
body compiles and runs as authored" *and* "the generator statically enumerates the
dependencies", with a classic `IIncrementalGenerator` and zero interceptors. The
reviewer's honest path holds. The only price is the static-analyzability
constraint (limitation 1–4), which is a property we *want* for deterministic,
agent-authored wiring — not a regression.
